## 🔑 Key Concepts

| Concept | What It Is |
|---|---|
| **S3** | Object storage service: stores files as objects inside buckets |
| **CloudFront** | AWS's CDN: caches and serves content from 450+ global edge locations |
| **Origin** | The source CloudFront pulls content from (here: the S3 bucket) |
| **Edge Location** | A data center in CloudFront's network closest to the requesting user |
| **Default Root Object** | The file CloudFront serves when the root URL is accessed |
| **Cache Invalidation** | Forces CloudFront to re-fetch content from origin instead of serving stale cache |
| **HTTPS/TLS** | Encryption in transit: CloudFront provides this automatically |
| **HTTP/2** | Faster multiplexed protocol, enabled on this distribution |
| **OAC (Origin Access Control)** | Restricts S3 access so only CloudFront can serve the file — bucket stays private |
| **Price Class** | Controls which edge locations are used: "All" gives the best global performance |

## 🚀 Setup Instructions

### Prerequisites
- An AWS account
- AWS CLI installed and configured
- A resume PDF ready to upload

---

### Step 1 — Create the S3 Bucket

```bash
aws s3 mb s3://perishaunak-resume --region us-east-2
aws s3 cp Peri_Shaunak_Resume.pdf s3://perishaunak-resume/
```

Keep **Block all public access** enabled — CloudFront handles access via OAC, so the bucket itself stays private.

---

### Step 2 — Create a CloudFront Distribution

```bash
aws cloudfront create-distribution \
    --origin-domain-name perishaunak-resume.s3.us-east-2.amazonaws.com \
    --default-root-object Peri_Shaunak_Resume.pdf \
    --enabled \
    --price-class PriceClass_All \
    --http-version http2
```

Or via the AWS Console: **CloudFront → Distributions → Create distribution**, set the origin to the S3 bucket, enable OAC, and set `Peri_Shaunak_Resume.pdf` as the default root object.

---

### Step 3 — Update S3 Bucket Policy

After creating the distribution, apply the CloudFront-generated OAC bucket policy so only CloudFront can access the S3 object:

1. Go to your S3 bucket → **Permissions** → **Bucket policy**
2. Paste the policy generated by CloudFront
3. Click **Save changes**

---

### Step 4 — Access the Resume

Once the distribution status changes to **Deployed**, the resume is live at:

```
https://d15b2yt2d869a0.cloudfront.net
```

## 🔄 Updating the Resume

Upload a new version to S3, then invalidate the CloudFront cache so the new file is served immediately:

```bash
# Upload new version
aws s3 cp Peri_Shaunak_Resume.pdf s3://perishaunak-resume/

# Invalidate CloudFront cache
aws cloudfront create-invalidation \
    --distribution-id E3BQD88HXG66FH \
    --paths "/*"
```

Without the invalidation, CloudFront will continue serving the old cached version until the TTL expires.

## 📖 What I Learned

- S3 stores files as objects in buckets — not a traditional filesystem, but scalable and durable object storage
- CloudFront acts as a caching layer in front of S3, reducing latency by serving users from the nearest edge location
- The **default root object** setting in CloudFront maps the bare URL directly to a specific file
- CloudFront provides HTTPS out of the box — no SSL certificate management required
- **Cache invalidation** is critical after updating files — without it, CloudFront continues serving the old cached version
- HTTP/2 enables multiplexed connections, improving load performance over HTTP/1.1
- OAC keeps the S3 bucket private while still allowing CloudFront to serve its contents publicly

---

## ⚠️ Known Limitations

- **Cache TTL**: CloudFront caches at edge locations. Always run an invalidation after uploading a new resume version.
- **Custom domain**: Currently using the default `.cloudfront.net` domain. A custom domain (e.g. `resume.shaunakperi.com`) requires an ACM SSL certificate and a CNAME DNS record.
- **No HTTP → HTTPS redirect**: Configure a CloudFront behavior to enforce HTTPS for best security practice.
